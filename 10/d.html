D問題：回数数列

一見、巨大な制約（k ≤ 109）に圧倒されそうな問題です。しかし、実際の出力パターンを注意深く観察することで、背後にある規則的な波が見えてきます。

★ 数列の背後に潜む規則性を可視化する ★

1. まずは愚直なシミュレーションコード

直前の項の値が「それまでに何回登場したか」を次の項にしていくルールです。配列と連想配列（map）を使い、問題文の指示通りに愚直にシミュレーションを行うコードを書いてみましょう。 

#include <bits/stdc++.h>
using namespace std;

bool solve() {
    int s, k;
    cin >> s >> k;
    if (s == 0 && k == 0) {
        return false;
    }
    vector<int> a;
    map<int, int> cnt;

    a.push_back(s);
    cnt[s] = 1;

    for (int i = 1; i < k; i++) {
        int nxt = cnt[a.back()];
        a.push_back(nxt);
        cnt[nxt]++;
    }

    cout << a[k - 1] << endl;
    return true;
}

int main() {
    while (solve());
}
⚠️ なぜこのコードでは時間切れ（TLE）になるのか？
このコードでは、第2項、第3項、…と順番に第 k 項まで1つずつ愚直に生成しています。つまり、ループはちょうど k − 1 回実行されるため、全体の計算量は O(k) になります。
今回の問題の制約は k ≤ 109 と非常に大きいため、最悪で10億回ものループが必要になります。一般的な競技プログラミングの制限時間（約2秒）で実行できるループ回数は 108（1億）回程度 が目安なので、この方法では全く間に合いません。

2. 法則性の発見

「10億回のシミュレーションができない」となれば、何らかの高速なジャンプ（数学的な規則性）が必要です。まずは小さな s について、シミュレーションプログラムを実行して得られた実際の出力（第40項まで）を眺めてみましょう。

【図①：各初期値 s における生成数列の実際の並び】

上の出力をじっくり観察すると、初期値である s が何度も繰り返し現れる ことに気がつきます（上の図の紫色の部分など）。
そこで、「s が出現するたびに数列を区切る」 という強力なアプローチを採用してみましょう。それぞれの区間が、明確な規則性を持った「群」として綺麗に分解されるはずです。

3. 群数列への分割と内部構造

例として s = 5 のとき、出現した 5 の位置で数列を綺麗に区切って並べ直すと、以下のような巨大な周期構造（群数列）が完成します。

【図②：群数列への分割と完全展開（s = 5 の例）】

このとき、第 i 群の数列の長さは以下のように結論付けることができます。

• i = 1 のとき： 1
• 1 < i ≤ s + 1 のとき： 2s − 2i + 3
➔ 【なぜ？】 前半の区間では、群が1つ進むたびに毎回「両端の2要素」が減っていくため、群の長さは規則的に 2ずつ短く なっていきます。
• s + 1 < i のとき： 2s
➔ 【なぜ？】 前半を抜けきると減る要素がなくなり、常に一定の周期（長さ 2s）でパターンが繰り返されるようになります。

さらに、第 i 群における第 j 項 (1 ≤ j ≤ 第 i 群の数列の長さ) の内部の値も、完全に数式化が可能です。

💡 1. i = 1 のとき

j = 1 のとき: s

💡 2. 1 < i ≤ s + 1 のとき

奇数番目の項は i − 1 から始まり 1 ずつ増加し、偶数番目は i − 1 で一定となるため、

• j が奇数のとき: (i − 1) + (j − 1) / 2
• j が偶数のとき: i − 1

💡 3. s + 1 < i のとき

奇数番目の項は常に i − 1 で一定、偶数番目の項は 1 から 1 ずつ増加するため、

• j が奇数のとき: i − 1
• j が偶数のとき: j / 2

4. 実装に向けたアドバイス

ここまで数式化できれば、与えられた k が「第何群の、第何項に含まれるか」を特定して答えを出すことができます。 実装する際は、以下のように前半ブロックと後半ブロックで分けて処理を考えることで、109という大きな制約を安全に突破できます。

1. 前半ブロック (1 < i ≤ s + 1) の処理

群の長さが徐々に短くなっていく区間です。ここで「 k は最大 109 だから、群を1つずつ調べるループを回すと TLE（時間切れ）になるのでは？」と心配になるかもしれません。 しかし、結論から言うと第 i 群に含まれるかを1群ずつ愚直に判定するループ処理で全く問題ありません。

なぜなら、群の長さは 2s−1, 2s−3, 2s−5... と減っていくため、i 群目までに消費する要素数は (2s−1) + (2s−3) + ... となり、群数が増えるほど二次関数的に爆発して増加します。 つまり、k 項に到達するまでに必要な群数は高々 √k 個程度 しかありません。 k = 109 の場合でも √109 ≈ 31,622 回程度のループで済むため、計算量は O(min(s, √k)) となり、実行時間制限に対して非常に余裕で間に合います。

無理に二次方程式の解の公式などをこねくり回すより、for 文などで「現在の群の長さを k から引いていく」処理を書く方が、実装ミスを防げて安全です。

2. 後半ブロック (s + 1 < i) の処理

前半のループを抜けてもまだ k が残っている（0より大きい）場合は、この「群の長さが 2s で完全に一定になる区間」に突入したことを意味します。 ここからはずっと同じ長さの群が続くため、もはや1つずつループで引いていく必要はありません。

1群の長さが常に 2s なので、「何群進んだか（商）」と「群の何番目か（余り）」は、残った k に対して、
• 進む群数: k / (2 * s)
• 群内の位置: k % (2 * s)
といった割り算を使うだけで、一発で位置を割り出すことができます。

5. 実装への3つのステップ

実装は大きく分けて以下の 3段階 で処理を行います。

① 第1群だけ特別扱いする

k = 1 のときは、初期値 s をそのまま出力して処理を終了します。

② 前半ブロックを1群ずつ処理する

ループを回し、現在の群の長さを k から引いていきます。k が群の長さ以下になれば、そこが答えの所属位置です。

③ 後半ブロックは割り算だけで処理する

残った k から、周期 2s を使った割り算で一撃で答えの値を逆算します。

6. 最終実装コード

以下のコードはそのままコピーして使用可能です。オーバーフローを防ぐため long long 型を使用しています。 

#include <bits/stdc++.h>
using namespace std;

bool solve(){
    long long s, k;
    cin >> s >> k;
    if (s == 0 && k == 0) return false;
    
    // ① 第1群だけ特別扱いする
    if (k == 1){
        cout << s << endl;
        return true;
    }
    
    k--; // 第1群(s)の1要素分を消費
    
    // ② 前半ブロックを1群ずつ処理する
    for (long long i = 1; i <= s; i++){
        long long t = 2 * s - 2 * i + 1; // 現在の群の長さ
        if (k <= t){
            if (k == t) cout << s << endl;
            else cout << (k % 2 ? k / 2 + i : i) << endl;
            return true;
        }
        else k -= t;
    }
    
    // ③ 後半ブロックは割り算だけで処理する
    long long p = (k - 1) / (2 * s);
    long long q = (k - 1) % (2 * s) + 1;
    cout << (q % 2 ? s + 1 + p : q / 2) << endl;
    return true;
}

int main(){
    while(solve());
}
